# 企画書：LLMO用コンテンツ生成システム

本資料は、データサイエンスインターンコーディング課題「課題2：LLMO用コンテンツ生成システム」における、システム設計および実装のアプローチをまとめた補足資料です。

---

## 1. 背景と着眼点（Focus）

従来の「検索エンジン向け（SEO）」のコンテンツ制作から、「生成AI向け（LLMO）」のコンテンツ制作へとパラダイムがシフトしています。
生成AIに「引用されたい」「参照されたい」記事を作るためには、単なるキーワードの羅列ではなく、**AIが情報を抽出しやすい構造的・高密度なフォーマット**が不可欠であると着眼しました。

具体的には、以下の4要素がLLMOにおいて重要であると定義しました。
1. **結論先行（Summary）**: AIは冒頭の情報を重視しやすいため、挨拶を省き結論から入る。
2. **厳密な定義（Definition）**: 「Xとは、〜である」という辞書的な構文を用いることで、AIが事実として抽出しやすくする。
3. **構造化データ（Table/List）**: 比較や要点をMarkdown表や箇条書きで整理し、論理関係を明確にする。
4. **疑問と回答（FAQ）**: ユーザーの検索クエリ（プロンプト）に直接合致しやすいQ&A形式を末尾に配置する。

---

## 2. 課題へのアプローチ（Approach）

上記の着眼点をシステムに落とし込むため、以下のプロセスで課題に取り組みました。

### ① 2段階生成フローの採用
「タイトル案の生成」と「記事本文の生成」を分離しました。
一括で生成するとLLMの出力意図がブレやすくフォーマットが崩れる原因となります。ユーザーが一度タイトルを選択（意図を固定）してから本文を生成することで、出力の安定性を高めました。

### ② プロンプトの厳密なテンプレート化
LLM（Gemini）が自由なフォーマットで出力してしまうのを防ぐため、システムプロンプトで「挨拶禁止」「指定の見出しの強制」を指示し、ユーザープロンプトで「Markdown見出しの下に何を書くか」を穴埋め形式で指定するアプローチをとりました。

### ③ ルールベースの「LLMOバリデータ」の実装
LLMの出力は確率的であり、プロンプトの工夫だけではフォーマット要件の完全な遵守を保証できません。
そこで、生成後に「Summaryが先頭にあるか」「定義構文が存在するか」「表があるか」「FAQがあるか」を正規表現等で機械的に検査し、スコア化してユーザーにフィードバックする仕組みを導入しました。

## 3. 工夫した点（Innovations）

実装において、単に動くものを作るだけでなく、**保守性・検証可能性・面接での説明可能性**を意識して以下の工夫を凝らしました。

- **Deterministic Stub（決定論的モック）の実装**
  - 開発初期から外部API（Gemini）に依存しきりになると、UI開発やテストの足枷になります。入力値からハッシュを計算し、常に固定のフォーマット済みMarkdownを返すローカルスタブを作成することで、API制限や通信待ちを気にせずフロントエンドの実装を高速化しました。
- **バリデーション結果の可視化とアクション提示**
  - 記事を出力して終わりではなく、UI上で「どの要件を満たし、どの要件を落としたか」を可視化しました。さらに「Markdown表を追加してください」などの具体的な改善アクション（Missing Actions）を提示するようにしました。
- **アーキテクチャの意思決定ログ（Decision Log）の運用**
  - なぜStreamlitを選んだのか（UI構築工数を下げてロジックに集中するため）、なぜ一括生成ではなく2段階なのかなど、開発中のトレードオフと判断を `docs/decisions.md` に記録しながら進めました。

## 4. 使用した技術（Tech Stack）

モダンで型安全、かつ迅速にプロトタイプを構築できる技術選定を行いました。

- **バックエンド**: `FastAPI`, `Pydantic`
  - 入出力のスキーマを厳格に定義し、APIの堅牢性を担保。
- **フロントエンド**: `Streamlit`
  - プロトタイプUIの爆速構築。セッションステートを活用したステップ遷移（生成→選択→表示）の実装。
- **AIインテグレーション**: `google-genai` (Gemini 2.5 Flash)
  - 高速かつコンテキストウィンドウの広いGeminiモデルを採用（最新の公式SDKを使用）。
- **開発ツール**: `uv` (パッケージ管理), `Ruff` (Linter/Formatter), `Pyright` (型検査)
  - 開発体験を向上させ、バグを未然に防ぐモダンなPythonエコシステムを構築。

## 5. 今後の展望（Next Steps）

期間とMVPのスコープの都合上見送りましたが、実運用に乗せる場合は以下の拡張を想定しています。

1. **自己修復（Self-Correction）ループ**: バリデータで不合格となった場合、そのフィードバックをプロンプトに含めてLLMに自動修正させる機能。
2. **Schema.org（JSON-LD）の自動生成**: 生成された記事から、構造化マークアップ用のJSON-LDを抽出して同時に出力する機能。
3. **複数モデル対応**: プロバイダの抽象化（`LLMClientProtocol`）は既に実装済みのため、OpenAIやAnthropicへの差し替え・比較機能の追加。
